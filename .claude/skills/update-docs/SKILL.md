# update-docs: ドキュメント一括更新

コミット履歴・差分をもとに、プロジェクト根幹ドキュメントを最新状態に同期するスキル。

## 手順

1. **変更内容の把握**
   以下を実行して更新対象を把握する。
   - `git log --oneline -20` — 最近のコミット履歴
   - `git diff HEAD~10..HEAD --stat` — 直近の変更ファイル一覧
   - 各ドキュメントの現在の内容を Read で確認

2. **README.md の更新**
   - プロジェクト概要・セットアップ手順・使い方が現状のコードと乖離していないか確認し、必要であれば更新する
   - `make bootstrap` / `make format` / `make lint` など、導入済みのコマンドが記載されているか確認する

3. **CLAUDE.md の更新**
   - ディレクトリ構成・アーキテクチャ方針・外部ライブラリ・コーディング規約・実装状況が現状と一致しているか確認し、必要であれば更新する
   - 実装が進んでいれば「現在の実装状況」セクションを更新する

4. **`.claude/settings.json` の更新**
   - 新たに追加されたコマンド許可・フック設定などが反映されているか確認し、必要であれば更新する

5. **`.claude/rules/` の更新（要確認）**
   - `Glob` で `.claude/rules/` 配下のファイルを確認し、現在のコードや設計と乖離していないか確認する
   - 新たに確立したコーディング規約・ファイル構成・レイヤー設計があれば追加を検討する
   - 更新・追加が必要と判断した箇所を特定し、**変更内容をユーザーに提示して確認を取る**
   - OKが出た箇所のみ更新する
   - 確認なしに `.claude/rules/` のファイルを編集・作成してはならない

6. **`Docs/` 配下の更新（要確認）**
   - 更新が必要と判断した箇所を特定し、**変更内容をユーザーに提示して確認を取る**
   - OKが出た箇所のみ更新する
   - 確認なしに Docs/ のファイルを編集してはならない

## 優先順位

更新の必要性が高い順：
1. 実装状況の変化（新機能・モジュール追加）
2. セットアップ手順の変化（ツール・コマンド追加）
3. アーキテクチャ・構成の変化
4. その他の記述の正確性

## 注意事項

- 内容に変化がないファイルは更新しない
- ドキュメントのスタイル・フォーマットは既存に合わせる
- 推測で書かず、コードや設定ファイルの実態に基づいて記述する
- `Docs/` は必ずユーザー確認を経てから更新する
